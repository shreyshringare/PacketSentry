## Problem Statement

PacketSentry has genuinely rare depth for a new-grad project: 7-model ensemble, from-scratch Aho-Corasick and GraphSAGE GNN, SHAP attribution on every alert, 241 tests at 92–100% coverage. But none of that is legible to a GitHub visitor in 10 seconds, and none of it is defensible in an ML interview without concrete benchmark numbers. The goal is to make the existing depth visible and interview-ready for ML/AI Engineer and Security/Infra Engineer roles.

**Target:** ML/AI Engineer + Security/Infra Engineer, new grad, strong portfolio (PacketSentry specializes, not carries).

---

## What Makes This Strong

- From-scratch GNN (no PyTorch Geometric) — genuinely rare at new-grad level
- SHAP on every alert — directly aligned with 2026 "explainability-first" interview trend
- FP feedback loop that auto-adjusts ensemble weights — shows production ML thinking
- Dual storage (DuckDB + ChromaDB) — demonstrates system design beyond just model building
- 241 tests, 92–100% coverage on core modules — engineering discipline signal

---

## Agreed Premises

1. The core depth (algorithms, models, tests) is already strong — the gap is visibility and metrics.
2. Without F1/recall numbers on NSL-KDD, ML decisions cannot be defended in an interview.
3. A web UI or demo GIF is required for GitHub visitors to understand what this does in under 10 seconds.
4. Docker + REST API moves this from "CLI tool" to "deployable ML system" — a key signal for ML roles.
5. For Security/Infra roles, the from-scratch algorithms are the main hook — already present.

## Approaches Considered

### Approach A: Metrics + Showcase (Phase 1)
NSL-KDD evaluation script, F1/precision/recall/ROC-AUC plots saved to `results/`, SHAP beeswarm visualizations, Jupyter notebook, model comparison table, demo GIF.
- Effort: M (2–3 days)
- Risk: Low — no architecture change

### Approach B: Web Dashboard + REST API + Docker (Phase 2)
FastAPI REST API + WebSocket wrapping DetectionPipeline, React/TypeScript web dashboard, Docker + demo GIF, NSL-KDD metrics integrated.
- Effort: L (4–6 days)
- Risk: Medium — new web layer

### Approach C: Full MLOps Pipeline
MLflow, model registry, Prometheus, GitHub Actions retraining CI, K8s manifests.
- Effort: XL
- Risk: High — too much surface area for a new-grad portfolio, hard to explain in 30 min

**Selected: A+B in phases.** Phase 1 ships in 2–3 days for immediate resume impact. Phase 2 adds the live demo URL. C deferred — adds no interview value vs B.

**Phase scope boundaries (hard):**
- Phase 2: Overview + Live Capture + Alert Detail (SHAP) + Memory/UMAP + Settings + REST API + WebSocket + Docker
- Phase 3 (deferred): Geo board, Replay screen, Training UI in Settings
- Timeline: 7–10 days solo (realistic)
- Rationale: Memory/UMAP added back — UMAP + ChromaDB + similarity search is the single most differentiating screen for ML/AI roles at new-grad level.
- Reason: Phase 2 at full scope = 1–2 weeks solo. MVP = 5–7 days.

### Screen 1 — Overview Dashboard

- Two YAxis

## Engineering Decisions (resolved)

**Auth:** Localhost-only for demo/portfolio. No auth required — this is a single-user local tool, not a multi-tenant service.

**DuckDB schema** (existing `alerts/store.py`, document for API layer):
```sql
alerts(id TEXT, timestamp FLOAT, severity TEXT, src_ip TEXT, dst_ip TEXT,
       dst_port INT, confidence FLOAT, rule TEXT, detectors TEXT,
       shap_json TEXT, embedding BLOB)
```
API queries: `WHERE severity=? AND timestamp > ?` — no new schema needed.

**Error handling:**
- WebSocket disconnect: client auto-reconnects with 2s backoff, max 10 attempts
- ip-api.com rate limit: cache IP→geo in memory dict (no repeat calls), fallback "Unknown" if rate-limited
- UMAP timeout: compute in `asyncio.run_in_executor` (background thread), return cached result while computing
- ChromaDB failure: log + return empty `similar_alerts: []`

**React Query strategy:**
- `/api/alerts`: 30s stale time, refetch on window focus
- `/api/flows/active`: 5s stale time (short-lived data)
- `/api/stats`: WebSocket-only, no polling
- Invalidate `alerts` query on `alert_event` WebSocket message

**Packet stream virtual scroll:** Use `react-window` (`FixedSizeList`), item height 22px, buffer last 500 lines in Zustand `captureStore.packets[]`.

**UMAP recompute debounce:** Queue new embeddings, recompute 30s after last arrival (not on every 50th — avoids burst recomputes).

**Ensemble weight normalization:** Sliders are independent. Live total shown. "Normalize" button divides each weight by total — user must click to apply. No auto-normalize on change (too surprising).

**WebSocket backpressure:** `ws_manager.py` maintains a per-client send queue (asyncio.Queue, maxsize=100). Slow clients are dropped after queue full, not broadcasted to synchronously.

**Performance targets:** Designed for 1–10k pps (home/lab network). Above 10k pps, packet stream slices to last 500 events client-side and WS messages are batched per 100ms tick.
